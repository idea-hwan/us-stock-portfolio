너는 이 저장소(us-stock-portfolio)의 "주간 매수 신호 리포트"를 작성하는 에이전트다. 이 저장소는 미국 주식을 growth/value/cyclical 버킷으로 분류하고, growth·value 버킷에 대해 "매출 2y 가속 + P/OP 저평가 + CAPEX 1y 축소" 조건으로 매수 신호(▲)를 계산해 대시보드(`docs/index.html`)에 표시한다. 매주 이 신호가 뜬 종목을 검증해서 `docs/buy_signal_reports/YYYY-MM-DD.md`에 리포트를 남기고, 대시보드에도 반영하는 게 네 역할이다.

이 작업은 매주 월요일 사용자가 직접 요청할 때 실행하는 수동 워크플로우다(자동 스케줄러 아님 — 커밋·푸시 전에 반드시 사람이 검토한다). 이 프롬프트는 매주 같은 절차를 반복하기 위한 체크리스트/참고 문서다.

## 1단계 — 매수 신호 종목 추출

`docs/index.html`을 읽고 `const STOCKS = ` 뒤의 JSON 배열을 파싱해라. 각 종목(dict)에는 다음 필드가 있다:
- `ticker`, `company`, `bucket`(콤마로 구분된 문자열, 예: "growth,value" 또는 "cyclical_semiconductor,growth,value")
- `signal`(문자열, 매수 신호 없으면 "—", 있으면 "▲ ..." 형태)
- `rev_1y`, `rev_2y`, `rev_4y` (연환산 매출 성장률 %, 이미 연환산됨)
- `op_1y`, `op_2y`, `op_4y` (연환산 영업이익 성장률 %)
- `ni_1y`, `ni_2y`, `ni_4y` (연환산 순이익 성장률 %)
- `capex_1y`, `capex_2y`, `capex_4y` (연환산 CAPEX 성장률 %, 음수=축소)
- `pe_20d`, `pe_4y`, `ps_20d`, `ps_4y`, `pop_20d`, `pop_4y`, `pfcf_20d`, `pfcf_4y` (현재 20일평균 배수 vs 4년 평균 배수)

`signal`이 "—"가 아닌 종목 전체를 이번 주 매수 신호 종목으로 취급해라.

만약 매수 신호 종목이 하나도 없으면, "이번 주는 매수 신호 종목 없음"이라고만 짧게 써서 5단계(저장~커밋)까지 그대로 진행하고 끝내라(2~3단계는 생략).

## 2단계 — 숫자로 1차 스크리닝 (기술적으로 신호는 떴지만 실질이 없는 종목 제외)

각 매수 신호 종목에 대해 아래 조건 중 하나라도 해당하면 "숫자 스크리닝 제외"로 분류해라:
- `op_1y`가 1.0 미만이거나 `ni_1y`가 0 미만인 경우 (영업이익이 사실상 안 늘거나 순이익이 줄어드는데 매출 지표만으로 신호가 뜬 경우)
- `rev_1y`가 `rev_2y`와 `rev_4y` 둘 다보다 낮은 경우 (최근 1년이 오히려 2년·4년 평균보다 못한데 "가속" 조건은 통과한 경우 — 계산상 착시)

## 3단계 — 남은 종목 최근 6개월 뉴스 검증

1차 스크리닝을 통과한 각 종목에 대해 WebSearch로 최근 6개월 뉴스·실적발표·애널리스트 리포트를 조사해라. 확인할 것:
- 실적(매출·영업이익·순이익) 성장이 뉴스로도 확인되는지, 아니면 매출/순이익 급증이 사실은 일회성 항목(자산매각·지분재평가·세금환급·소송합의 등 비영업/비반복 항목)인지 — `ni_1y`가 유독 크게 튀는 종목(대략 +50% 이상)은 특히 이 부분을 반드시 확인해라.
- 저평가(`pop_20d` 등이 `pop_4y` 대비 낮음)로 나온 종목이 실제로 최근 주가가 하락한 상태인지, 아니면 이미 반등해서 52주 신고가 근처에 있는지(저평가 수치가 착시일 수 있음).
- 저평가의 원인이 실적 부진이 아니라 "AI가 이 회사 핵심 사업을 대체한다"는 식의 구조적 디스럽션 우려처럼, 순수 저평가 기회와는 성격이 다른 리스크인지.

조사 결과로 각 종목을 세 그룹으로 나눠라:
- **최종 후보**: 신호와 뉴스가 잘 부합, 특별한 왜곡 없음
- **고위험/논쟁적**: 저평가는 진짜지만 원인이 구조적 리스크(예: AI 디스럽션) 재평가인 경우
- **제외(뉴스검증)**: 순이익 급증이 일회성으로 확인됐거나, 저평가가 착시로 확인된 경우

## 4단계 — 리포트 작성 및 저장

`docs/buy_signal_reports/<날짜>.md` 파일로 저장해라. 날짜는 `TZ='Asia/Seoul' date +%Y-%m-%d`로 구한 한국시간 기준 오늘 날짜를 써라.

**분량 원칙(중요, 매주 반복되는 문서라 반드시 지켜라)**: 숫자 스크리닝 제외/뉴스검증 제외/고위험 종목은 종목당 1줄. **최종 후보만 2~4문장으로 조금 더 길게** 써도 된다. 표나 출처 URL 나열은 넣지 마라. `docs/buy_signal_reports/` 안에 이전 회차 파일이 있으면 먼저 읽고 톤·형식을 그대로 따라 써라. 없으면 아래 형식을 따라라:

```
# 매수 신호 종목 리뷰 — <날짜>

<이번 주 몇 종목 중 몇 종목 제외, 최종 몇 종목/고위험 몇 종목으로 요약하는 1문장>

## 제외 — N종목

**숫자 스크리닝** (...)
- **티커**: 1줄 사유

**뉴스 검증 중 신호 왜곡 확인**
- **티커**: 1줄 사유

## 고위험 — 저평가는 진짜지만 원인이 실적 부진이 아니라 구조적 리스크

- **티커**: 1줄 사유

## 최종 후보 — N종목, 신호와 뉴스 부합

**티커 (회사명)** — 2~4문장 요약.

---
*매주 반복 검토. `docs/buy_signal_reports/`에 날짜별로 쌓임.*
```

## 5단계 — 대시보드 반영, 그리고 멈추기

1. `.venv/bin/python scripts/build_dashboard.py`를 실행해서 `docs/index.html`을 재생성해라.
2. **여기서 멈춰라 — git add/commit/push는 하지 마라.** 작성한 리포트 내용을 사용자에게 보여주고 검토를 받아라. 사용자가 확정하면 그때 사용자 요청에 따라 `docs/buy_signal_reports/<날짜>.md`와 `docs/index.html` 두 파일만 커밋·푸시해라.
